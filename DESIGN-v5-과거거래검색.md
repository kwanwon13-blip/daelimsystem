## 한 줄 요약

**"단가표를 만드는" 접근을 버리고, 몇 년치 과거 제작 매출 데이터를 검색 가능한 형태로 만들어서 직원이 견적 작성 중에 "이 거래처에 과거 비슷한 조합으로 얼마에 나갔는지"를 즉시 볼 수 있게 한다.**

---

## 문제 정의

### 표면적 문제
직원이 견적을 낼 때마다 거래처별 단가를 몰라서 대표에게 물어본다.

### 진짜 문제
- 제작품은 재질 × 두께 × 가공 × 부자재 × 규격 × 거래처 조합이 무한대
- 거래처별 획일화된 단가표가 존재하지 않음 (만들 수도 없음)
- 대표 머릿속 패턴 매칭이 유일한 가격 결정 로직
- 문서화하려는 모든 시도(리바트 매핑 v1~v4)가 실패
- 사전 구축 방식(기본 단가 + 옵션 조합)은 재료 다양성 앞에서 붕괴

### 실패한 과거 접근
1. **v1 — 엑셀식 평탄화 UI**: 데이터 모델 문제를 UI로 못 풀었음
2. **v2 — 과거 견적 복사**: 제작품이라 "같은 조합"이 거의 안 나와 재사용 실패
3. **v3 — 원가 계산기(요소 분해)**: 재료/가공 다양성이 너무 넓어 분해 불가능
4. **v4 — 결재 시스템 연동 점진 축적**: 처음부터 쌓기 → 6개월간은 여전히 대표 의존

### v5로 전환한 결정적 이유
**몇 년치 제작 매출 데이터가 이미 엑셀로 존재한다.** 즉 대표 머릿속 지식이 이미 "실제 거래 기록" 형태로 외부화되어 있음. 단지 검색 가능한 DB가 아닐 뿐. 이건 사전 세팅 노동이 거의 0이라는 뜻.

## 데이터 현황 (검증 완료)

라코스 파일(`라코스 매출가 확인자료.xlsx`) 실측 결과:

### 컬럼 구조
```
발행일자 | 품명 | 규격 | 매출수량 | 단가 | 금액 | 비고 | 납품처 | ...
```

### 실제 샘플
```
2025-06-20 | 3t포맥스+6개 타공           | 420*297       | 464 | 5,000   | 2,320,000
2025-03-28 | 3t포맥스+사방타공+양면테이프 | 210*297       | 115 | 1,300   |   149,500
2025-03-28 | 3t포맥스+사방타공+양면테이프 | 450*600       | 260 | 5,400   | 1,404,000
2025-07-01 | 락카보관함                  | 500*450*1050  |   1 | 350,000 |   350,000
2025-06-20 | 고휘도반사스티커             | 140*500       | 500 | 1,750   |   875,000
```

### 핵심 관찰
- **품명 필드**는 `+` 구분자로 반쯤 구조화되어 있음 → 토큰화 가능
- **규격 필드**는 `*` 구분자로 치수 분리 가능 (2D/3D 자동 판별)
- **두께는 `\d+t` 패턴**으로 정규식 추출 가능
- **재질 사전**(포맥스/아크릴/MDF/철판 등)으로 매칭 가능
- **같은 품목도 규격에 따라 단가가 크게 다름** → 거래처 × 품목속성 × 규격 3차원 룩업이 본질
- 잡음 행(운반비, 외상매출금, 전표합계)은 키워드 필터로 제거

### 파싱 난이도: 낮음
`+` 구분자 + `*` 구분자 + 재질 사전 매칭만으로 80%+ 자동 구조화 가능.
나머지 20%(완제품명 단일 토큰, 특수 케이스)는 완제품 전체 이름을 그대로 키로 사용.

## 시스템 설계

### 스코프
- **포함**: 제작물 견적만
- **거래처**: 현재 등록된 매출처 (소수)
- **제외**: SM 안전용품, 일반 판매

### 아키텍처

```
┌────────────────────────────────────────┐
│   거래처별 매출 엑셀 (라코스, 리바트,    │
│   광고천하, 현대건설, 삼성SDI 등)        │
└──────────────┬─────────────────────────┘
               │ (1) 1회성 + 월별 적재
               ▼
┌────────────────────────────────────────┐
│   SQLite: sales_history                 │
│   - 원본 행 그대로 보존                  │
│   - 파싱된 속성 컬럼 추가                │
└──────────────┬─────────────────────────┘
               │ (2) 유사도 검색 API
               ▼
┌────────────────────────────────────────┐
│   견적 작성 UI                          │
│   ┌──────────┐  ┌───────────────────┐ │
│   │ 입력 폼   │  │ 과거 유사 거래     │ │
│   │           │◀─│ (3~5건 자동 표시)  │ │
│   └──────────┘  └───────────────────┘ │
└────────────────────────────────────────┘
```

### 데이터 모델

```sql
CREATE TABLE sales_history (
  id INTEGER PRIMARY KEY,

  -- 원본 필드 (그대로 보존)
  vendor TEXT NOT NULL,              -- 거래처 (파일별 또는 납품처 컬럼)
  sale_date DATE NOT NULL,           -- 발행일자
  raw_name TEXT NOT NULL,            -- 품명 원본
  raw_spec TEXT,                     -- 규격 원본
  qty INTEGER,                       -- 수량
  unit_price INTEGER NOT NULL,       -- 단가
  amount INTEGER NOT NULL,           -- 금액
  note TEXT,                         -- 비고

  -- 파싱된 속성 (검색용)
  material TEXT,                     -- 포맥스/아크릴/MDF/철판/...
  thickness_mm REAL,                 -- 3, 5, 10 (null 가능)
  processings TEXT,                  -- JSON 배열: ["사방타공","양면테이프"]
  accessories TEXT,                  -- JSON 배열
  width_mm INTEGER,                  -- 규격 W
  height_mm INTEGER,                 -- 규격 H (또는 D)
  depth_mm INTEGER,                  -- 3D일 때만
  is_finished_product BOOLEAN,       -- "락카보관함" 같은 완제품 여부
  product_key TEXT,                  -- 완제품이면 이름 그대로, 아니면 속성 조합

  -- 메타
  source_file TEXT,                  -- 원본 엑셀 파일명
  source_row INTEGER                 -- 원본 행 번호
);

CREATE INDEX idx_vendor_product ON sales_history(vendor, product_key);
CREATE INDEX idx_material_thick ON sales_history(material, thickness_mm);
CREATE INDEX idx_date ON sales_history(sale_date DESC);
```

### 파싱 규칙 (품명 → 속성)

입력: `3t포맥스+사방타공+양면테이프`

1. `+`로 split → `["3t포맥스", "사방타공", "양면테이프"]`
2. 첫 토큰에서:
   - `\d+t` 매칭 → thickness_mm = 3
   - 남은 문자열 = `포맥스` → 재질 사전 매칭 → material = "포맥스"
3. 나머지 토큰 분류:
   - 가공 사전(`타공`, `절곡`, `용접`, `재단` 등 포함) → processings
   - 부자재 사전(`양면테이프`, `볼트`, `스티커` 등 포함) → accessories
4. 사전에 안 맞으면 → `is_finished_product = true`, product_key = 원본 품명

입력: `락카보관함`
→ `+` 분리 실패 + 재질 키워드 없음 → 완제품으로 분류, product_key = "락카보관함"

### 규격 파싱

- `420*297` → width=420, height=297 (2D)
- `500*450*1050` → width=500, depth=450, height=1050 (3D)
- 비어있거나 다른 형식 → null 저장 (검색 시 치수 가중치 0)

### 사전 초기화
재질/가공/부자재 사전은 파싱 시작 전에 **전체 데이터 품명을 모두 토큰화 → 빈도 순 상위 30개 → 수동 분류** 방식으로 한 번에 만듭니다. 1시간 정도 작업.

### 잡음 필터
다음 행은 적재에서 제외:
- 품명에 `운반비`, `외상매출금`, `입금`, `출금`, `전표합계` 포함
- 품명이 null
- 단가가 0

## 유사도 검색 API

### 엔드포인트
```
POST /api/sales-history/similar
Body: {
  vendor: "라코스",
  material: "포맥스",
  thickness_mm: 3,
  processings: ["사방타공"],
  width_mm: 420,
  height_mm: 297,
  limit: 5
}
```

### 유사도 스코어 계산

```
score =
  (거래처 일치)      * 10  +
  (재질 일치)        * 5   +
  (두께 일치)        * 4   +
  (가공 교집합 수)   * 3   +
  (부자재 교집합 수) * 2   +
  (면적 근사도)      * 2   +
  (최근성 가점)      * 1
```

- 거래처 일치 가중치를 가장 크게 (같은 거래처에 같은 조건으로 나간 가격이 가장 강한 신호)
- 면적 근사도: `max(0, 1 - |A1-A2|/max(A1,A2))`
- 최근성: 최근 6개월 내면 +1, 1년 이내면 +0.5
- 정확히 같은 조합이 여러 번 있으면 모두 표시 (평균 내지 않음 — 판단은 직원이)

### 응답
```json
{
  "matches": [
    {
      "vendor": "라코스",
      "raw_name": "3t포맥스+사방타공+양면테이프",
      "raw_spec": "420*297",
      "unit_price": 5000,
      "qty": 464,
      "sale_date": "2025-06-20",
      "score": 24.3,
      "why": "거래처·재질·두께·가공 일치, 규격 근사도 95%"
    },
    ...
  ]
}
```

## 성공 지표

**1차 (즉시 측정 가능)**
- 파싱 성공률 ≥ 80%
- 유사 거래 검색 응답 ≤ 200ms
- 견적 작성 시 과거 거래가 상위 3개 안에 "쓸만한 것"이 있는 비율 ≥ 70%

**2차 (1달 사용 후)**
- 직원이 대표에게 단가 질문하는 빈도 **50% 이상 감소**
- 견적 작성 평균 소요 시간 감소
- 직원이 독립적으로 마감한 견적 비율 증가

**실패 신호**
- 파싱 성공률 < 60% → 사전 재설계 필요
- 유사 거래가 있어도 직원이 결국 대표 확인 → 가중치 재튜닝 또는 근본적 재검토

## 리스크 & 대응

**리스크 1: 품명 표기가 파일/시기마다 다름**
- 대응: 정규화 규칙을 파싱 사전에 누적. 첫 적재 때 로그 남기고 수동 보정 1회.

**리스크 2: 거래처가 새로 생기면 과거 데이터 0**
- 대응: 유사도 검색에서 거래처 가중치를 약간 낮춰 "다른 거래처의 유사 조합"을 fallback으로 보여줌. 여전히 참고값 역할.

**리스크 3: 과거 단가가 현재 원가 상승을 반영 못함**
- 대응: 최근성 가점 + UI에 날짜 명시. 직원이 오래된 거면 자연스럽게 조정. 필요하면 재료별 "시세 조정률" 필드 추가 (나중에).

**리스크 4: 대표가 "그건 옛날 가격이야" 하면?**
- 대응: 견적 저장 시 해당 거래는 새 데이터로 추가됨. 자연스럽게 최신 데이터가 우선순위 가짐. 의도적 업데이트 UI는 필요 없음.

**리스크 5: SM 안전용품 쪽에서 "우리도 써야겠다" 하면 스코프 터짐**
- 대응: 현재 스코프는 제작물 고정. SM은 후속 프로젝트로 분리. 스코프 확장 요청이 오면 이 설계 문서에서 재검토 시작.

## 열린 질문 (구현 시 결정)

1. **파싱 사전을 어디에 둘 것인가** — JSON 파일 vs DB 테이블. DB 테이블이 관리 UI 붙이기 쉬움.
2. **거래처명 정규화** — `(주)삼성라코스-021` vs `라코스` 같은 변형 처리. 초기엔 수동 매핑 테이블.
3. **수량별 할인 학습** — 같은 품목 대량/소량 단가가 다를 때 자동 감지할지, 그냥 사례 나열만 할지. v1은 나열만.
4. **대표 결재 문의 시 자동 태그** — 직원이 문의하는 속성을 결재 본문에 자동 첨부. 대표가 답할 때 그 데이터가 곧 신규 sales_history로 들어갈 수 있도록 훅.

## 핵심 철학

이 설계의 본질은 이겁니다:

> **대표 머릿속 지식을 문서화하려 하지 말고, 대표가 이미 내린 결정(과거 거래)을 검색 가능하게 만든다.**

사전 구축 노동은 0에 가깝습니다. 리바트 매핑 같은 고통은 다시 반복하지 않습니다. 대신 이미 존재하는 수천 건의 실제 거래가 곧 단가표가 됩니다. 완벽한 자동화가 아니라 **"대표에게 물어봐야 할 빈도를 10분의 1로 줄이는"** 현실적 목표.

제작품의 세계는 표로 안 눌러집니다. 그래서 표를 만들지 않습니다. 대신 검색합니다.
