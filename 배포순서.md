# 배포 순서 (실수 방지용)

## Codex 자동 배포 방식 (새 채팅 기준)

새 채팅에서 "푸시풀", "배포", "서버 반영"이라고 하면 아래 순서로 진행한다.

1. 로컬 수정 확인
   ```powershell
   git status --short --branch
   git diff
   ```
   관련 없는 변경이나 미추적 파일은 stage하지 않는다.

2. 서버는 `origin/main`을 pull하므로, 배포 커밋은 `main` 기준으로 만든다. 현재 작업 브랜치가 `main`이 아니거나 밀면 안 되는 커밋이 섞여 있으면 임시 worktree를 쓴다.
   ```powershell
   git worktree add C:\tmp\price-list-app-main-deploy origin/main
   ```

3. 필요한 파일만 커밋하고 GitHub `main`으로 push한다.
   ```powershell
   git add -- 수정한파일
   git commit -m "커밋 메시지"
   git push origin HEAD:main
   ```

4. 서버 PC의 실제 `.env`에서 `CONTROL_DAEMON_SECRET`을 읽어 서버 pull API를 호출한다. secret 값은 화면에 출력하지 않는다.
   ```powershell
   $envPath='\\192.168.0.133\DD\price-list-app\.env'
   $line = Get-Content -LiteralPath $envPath | Where-Object { $_ -match '^CONTROL_DAEMON_SECRET=' } | Select-Object -First 1
   $secret = (($line -replace '^CONTROL_DAEMON_SECRET=', '').Trim())
   curl.exe -s -X POST -H "X-Control-Secret: $secret" -H "Content-Type: application/json" -d "{}" http://192.168.0.133:3000/api/git-pull
   ```

5. 서버를 재시작하고 반영을 확인한다.
   ```powershell
   curl.exe -s -X POST -H "X-Control-Secret: $secret" http://192.168.0.133:3002/restart
   Start-Sleep -Seconds 8
   curl.exe -s http://192.168.0.133:3000/api/version
   ```

`POST /api/git-pull`이 `Unauthorized`이면 거의 항상 서버 실제 `.env`의 secret을 쓰지 않은 것이다. 로컬 `.env` 값과 서버 값은 다를 수 있다.

## STEP 1 — 💻 내 컴터에서

**`git-setup.bat` 실행**

→ 코드가 GitHub에 올라감
→ DB 파일은 안 올라감 (정상, .gitignore로 막혀있음)

---

## STEP 2 — 🖥 서버에서

**① `git-pull-server.bat` 실행**

→ GitHub에서 최신 코드 받아옴 + 서버 자동 재시작

**② `node migrate-variants.js` 실행** (터미널에서)

```
cd D:\price-list-app
node migrate-variants.js
```

→ 서버 DB에 단가 반영 (기존 데이터 안 날아감)

**③ 서버 재시작** (migrate 실행 후 필수)

→ `서버중지.bat` 실행 후 `테스트서버실행.bat` 실행
→ 브라우저 F5 새로고침

---

## ⚠️ 절대 하면 안 되는 것

| 하면 안 됨 | 이유 |
|-----------|------|
| 서버에서 `git-setup.bat` 실행 | 서버 코드가 GitHub를 덮어씌움 |
| 내 컴터에서 `node migrate-variants.js` | 내 컴터 DB만 바뀜, 서버엔 반영 안 됨 |
