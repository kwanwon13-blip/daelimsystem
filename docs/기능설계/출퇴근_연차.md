## 0. 직원 마스터 (핵심 전제)

### 왜 필요한가
- CAPS는 이름 기준으로 출퇴근 데이터를 보냄
- ERP 계정(users[])이 없는 직원(예: 안소현)은 현재 누락됨
- **직원 마스터(employees[])를 별도로 관리해야 함**

### 데이터 구조 (조직관리.json에 employees[] 추가)
```js
// 조직관리.json
{
  "users": [ /* ERP 로그인 계정 — 기존 그대로 */ ],
  "departments": [ /* 부서 — 기존 그대로 */ ],
  "employees": [   // ← 신규 추가: 모든 재직 직원
    {
      "id": "emp_001",
      "이름": "안소현",
      "부서": "디자인팀",
      "직위": "대리",
      "입사일": "2023-03-01",
      "퇴사일": null,          // null = 재직중
      "근태규칙": "general",   // general | 영업지원 | 영업관리 | 제외
      "capsName": "안소현",    // CAPS에서 오는 이름 (다를 경우 매핑용)
      "userId": null           // ERP 계정 연결 (없으면 null)
    }
  ]
}
```

### 규칙
- **출퇴근/연차 관리는 employees[] 기준**으로 동작
- users[]는 ERP 로그인 계정 관리용 (기존 유지)
- `userId`로 두 목록 연결 (ERP 계정 있는 직원만 연결됨)
- `퇴사일`이 있으면 출퇴근/연차 목록에서 자동 제외

### 수정 파일
| 파일 | 작업 |
|------|------|
| `db.js` | jsonStores의 조직관리 초기값에 `employees: []` 추가 |
| `server.js` | GET/POST/PUT /api/employees 엔드포인트 추가 |
| `public/index.html` | 사용자 관리 탭에 "직원 목록" 섹션 추가 |

### server.js API
```js
// 전체 직원 목록 (재직자만)
app.get('/api/employees', requireAuth, (req, res) => {
  const org = db['조직관리'].load();
  const active = (org.employees || []).filter(e => !e.퇴사일);
  res.json(active);
});

// 직원 추가
app.post('/api/employees', requireAdmin, (req, res) => {
  const org = db['조직관리'].load();
  if (!org.employees) org.employees = [];
  const emp = { id: `emp_${Date.now()}`, ...req.body, 퇴사일: null };
  org.employees.push(emp);
  db['조직관리'].save(org);
  res.json(emp);
});

// 직원 수정 (부서/직위/근태규칙/퇴사일 변경)
app.put('/api/employees/:id', requireAdmin, (req, res) => {
  const org = db['조직관리'].load();
  const idx = org.employees.findIndex(e => e.id === req.params.id);
  if (idx === -1) return res.status(404).json({ error: '직원 없음' });
  org.employees[idx] = { ...org.employees[idx], ...req.body };
  db['조직관리'].save(org);
  res.json(org.employees[idx]);
});
```

## 1. 데이터 흐름

```
ADT캡스 → 브릿지(자동) → 우리 DB 저장 → 자동분류 → 화면 표시
                                                ↓
                                        이상건 강조 표시
                                                ↓
                                        관리자 확인/수정
                                                ↓
                                        확정 → 엑셀 다운로드
```

### 1단계: 캡스 데이터 수신
- 브릿지가 실시간으로 캡스에서 데이터 가져옴
- 받는 데이터: **이름, 날짜, 출근시간, 퇴근시간** (4가지)

### 2단계: DB 저장
- 캡스에서 가져온 원본 데이터를 우리 DB에 저장
- 한번 저장되면 캡스 연결이 끊겨도 데이터 유지
- 캡스 원본은 수정 불가 (별도 필드로 관리자 수정사항 저장)
- 관리자가 시간 수정 시: 원본 보존 + 수정값 별도 저장 (현재 코드와 동일)
- 수정된 셀에 마우스 올리면 **"원본: 09:15 → 수정: 09:00"** 툴팁 표시

### 3단계: 자동분류
- 출퇴근 기록을 바탕으로 자동 판단:
  - **정상**: 출근 ≤ 08:30, 퇴근 기록 있음
  - **지각**: 출근 > 08:30
  - **출근 미기록**: 퇴근만 있고 출근 기록 없음
  - **전체 미기록**: 평일인데 출퇴근 기록 모두 없음
  - **오전반차 추정**: 13:30~14:30 사이 출근
  - **오후반차 추정**: 11:00~12:00 사이 퇴근

### 4단계: 상태 표시
| 상태 | 색상 | 의미 |
|------|------|------|
| **확인필요** | 🟠 주황 배경 + 굵은 글씨 | 미기록/자동분류된 건. 관리자가 확인해야 함 |
| **지각** | 🔴 빨간 글씨 | 08:30 이후 출근. 자동 표시, 별도 확인 불필요 |
| **정상** | ⬜ 일반 | 출퇴근 모두 정상 기록됨 |
| **확정됨** | ✅ 초록 체크 | 관리자가 확인/수정 완료한 건 |

### 5단계: 관리자 확인/수정
- 주황색(확인필요) 항목을 클릭하면 선택지:
  - 연차
  - 오전반차
  - 오후반차
  - 출퇴근 누락 (정상처리)
  - 기타 (메모 입력)
- 선택하면 주황색 해제 → 확정 상태로 변경

### 6단계: 화면에서 확인
- 확정된 데이터는 화면에서 바로 확인 (별도 엑셀 다운로드 불필요)


## 2. 부서별 근무규칙

| 부서 | 규칙 | 설명 |
|------|------|------|
| **일반** (디자인팀, 경영지원 등) | 기본 | 출근 08:30, 지각 체크, 미기록 확인필요 |
| **영업지원** (납품팀) | 유연근무 + 토요근무 | 조퇴 없음. 토요일 출근 시 정상근무 인정 |
| **영업관리** | 근태면제 | 퇴근만 찍혀도 정상. 지각/미기록 체크 안 함 |
| **남관원** | 기록 제외 | 출퇴근 기록부에서 완전히 제외 |


## 3. 연차 관리

### 3-1. 데이터 구조 (기존 엑셀에서 이관)

#### DB_휴가사용 (1,342건)
- 성명, 부서, 직위, 휴가항목(연차/반차/외출/경조휴가/예비군 등), 사용일자, 연차일수, 사유
- 출퇴근 기록에서 관리자가 "연차/반차" 확정하면 여기에 자동 추가

#### DB_연차추가공제 (177건)
- 성명, 부서, 직위, 적용연도, 추가일수, 공제일수, 사유
- 예: 임신휴가 추가 15일, 마감 공제 등

#### DB_연차수당지급 (20건)
- 성명, 지급일, 연차수당(일), 정산일수, 지급액

### 3-2. 연차 계산 기준

#### ① 회계년도 기준 (평소 화면에 표시)
- 1월 1일 ~ 12월 31일 기준
- 입사 첫해: 매월 1일씩 발생 (최대 11일)
- 2년차부터: 15일
- 3년차부터: 매 2년마다 +1일 (최대 25일)

#### ② 입사일 기준 (퇴사정산 시에만 사용)
- 입사일 ~ 입사일+1년 기준
- 1년 미만: 매월 1일씩 (최대 11일)
- 1년 이상: 15일
- 3년 이상: 매 2년마다 +1일 (최대 25일)
- 퇴사 시 회계년도/입사일 기준 중 잔여가 더 많은 쪽으로 정산

### 3-3. 연차 현황 화면 (회계년도 기준)
- 직원별 한 행:
  - 이름, 부서, 직위, 입사일
  - 연차사용기간 (예: 2026-01-01~2026-12-31)
  - 발생연차 → 추가 → 공제 → 총연차
  - 사용분 → 잔여
  - **월별 사용현황** (1~12월 각각 몇 개 썼는지)
- 이름 클릭 → 개인별 사용내역 상세 (날짜, 유형, 사유)
- 연차수당 관련 컬럼은 제외

### 3-4. 연차 추가/공제 관리
- 관리자가 직접 추가/공제 입력 (사유 필수)
- 예: 임신휴가 +15일, 전년도 이월 공제 등

### 3-5. 연차촉진통지서 (자동 발송)
- 관리자가 기간 설정 + 버튼 클릭
- 시스템이 미사용 연차가 많은 대상자 자동 추출
- 법정 양식(연차촉진통지서) 자동 생성
- 공지사항처럼 해당 직원에게 알림 발송
- 직원이 사용계획서를 시스템에서 제출 가능

### 3-6. 연차정산서 (퇴사 시)
- 회계년도 기준 vs 입사일 기준 나란히 비교
- 연차별(1년차, 2년차, ...) 발생/사용/잔여 상세
- 더 많이 남은 기준으로 최종 정산


## 4. 화면 UI 방향

### 현재 문제점
- 자동분류와 확정의 구분이 불명확
- 이상건이 눈에 잘 안 띔
- 너무 많은 정보가 한 화면에 있음

### 개선 방향
- **주황색 강조**: 확인필요 항목은 배경색 + 굵은 글씨로 확실히 구분
- **상태 필터**: "확인필요만 보기" 필터로 미처리 건만 모아보기
- **간결한 카드**: 핵심 정보(이름, 출근, 퇴근, 상태)만 표시
- **클릭 → 즉시 처리**: 주황색 항목 클릭 → 드롭다운에서 바로 선택 → 확정


## 5. 작업 순서 (제안)

### Phase 1: 출퇴근 기록부
1. **캡스 데이터 → DB 저장** 구조 확립
2. **자동분류 + 상태 표시** (주황/빨강/정상/확정)
3. **관리자 확인/수정** 기능 + 원본비교 툴팁
4. **부서별 규칙** 적용

### Phase 2: 연차 관리
5. **기존 엑셀 데이터 이관** (DB_휴가사용 1,342건 + DB_연차추가공제 177건 + DB_연차수당지급 20건)
6. **연차 현황 화면** (회계년도 기준, 월별 사용현황, 개인별 상세)
7. **연차 추가/공제** 관리 기능
8. **연차촉진통지서** 자동 발송 (기간 설정 → 대상자 추출 → 공지 알림)

### Phase 3: 정산
9. **연차정산서** (퇴사 시 회계년도/입사일 기준 비교 정산)
